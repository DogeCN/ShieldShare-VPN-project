# Quick Start Guide: ShieldShare Load Testing

## Essential Metrics Summary

Beyond throughput and latency, you should measure:

### Critical Metrics
1. **Connection Metrics**: Establishment time, success rate, concurrent capacity
2. **Resource Utilization**: CPU, memory, battery per client
3. **Quality of Service**: Packet loss, jitter, fairness, TTFB
4. **Scalability**: Throughput/latency degradation with client count
5. **Reliability**: Error rates, timeout rates, recovery time
6. **Protocol Performance**: HTTP vs HTTPS overhead, keep-alive efficiency

See `TESTING_STRATEGY.md` for complete list.

## Testing Strategy: Physical vs Virtual Clients

### Strategy
- **1 client**: 1 physical device (real device)
- **5 clients**: 5 physical devices (1 real device each)
- **10, 30, 60 clients**: All virtual clients (Docker containers)
- Each virtual client gets unique IP address via Docker network namespaces
- No physical devices needed for 10+ client tests

### Setup Steps

1. **Install Docker** on your test machine(s)
   ```bash
   # Mac
   brew install docker docker-compose
   
   # Linux
   sudo apt-get install docker.io docker-compose
   ```

2. **Build Test Client Image**
   ```bash
   cd tests/load-testing
   ./setup.sh
   ```

3. **Configure Network**
   - Edit `docker-compose.yml` to set correct proxy host IP
   - Each client gets unique IP from subnet (e.g., 192.168.43.101, 192.168.43.102, ...)
   - Adjust subnet if your hotspot uses different range

4. **Run Tests**
   ```bash
   # For 1 client: Use 1 physical device manually
   # For 5 clients: Use 5 physical devices manually
   
   # For 10+ clients: All virtual
   python test-orchestrator.py --clients 10 --duration 900 --proxy-host 192.168.43.1
   python test-orchestrator.py --clients 30 --duration 1200 --proxy-host 192.168.43.1
   python test-orchestrator.py --clients 60 --duration 1800 --proxy-host 192.168.43.1
   ```

5. **Collect Results**
   - Results saved to `test-results/` directory
   - All results automatically aggregated by orchestrator

## Ensuring Different IP Addresses

### Method 1: Docker Network (Recommended)
Each container gets unique IP from configured subnet:
```yaml
networks:
  testnet:
    ipam:
      config:
        - subnet: 192.168.43.0/24
```

### Method 2: Network Namespaces (Linux)
```bash
sudo ip netns add client1
sudo ip addr add 192.168.43.101/24 dev veth1
```

### Method 3: Static IP Assignment
Configure your router/hotspot to assign static IPs based on MAC addresses.

## Common Issues

### Clients Can't Connect
- Verify proxy server is running
- Check firewall settings
- Ensure correct IP address
- Test with single client first

### Docker Network Issues
```bash
# Check network
docker network ls
docker network inspect shieldshare-test_testnet

# Restart network
docker-compose down
docker-compose up -d
```

### Resource Exhaustion
- Reduce concurrent connections per client
- Run fewer clients per machine
- Increase test machine resources
